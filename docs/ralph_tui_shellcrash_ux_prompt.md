使用 Ralph 循环，基于现有代码库和 TUI 设计技能，大幅激进,可丢弃legacy来重构和优化 `shellcrashgo` 的终端 UI / UX，同时保持行为稳定、安全、可维护。

---

## 任务目标

- 让 `shellcrashgo` 的整体交互体验更符合现代 TUI / CLI 最佳实践：可预测、键盘友好、可配置、易脚本化。
- 在不破坏已有核心功能的前提下，重构与 UI / UX 相关的代码结构、状态设计和交互逻辑，使其更易理解、更安全、更易扩展。
- 全程遵守本仓库的开发约定（尤其是 **不触碰真实网络栈** 的要求，仅做“文件级 / 逻辑级验证”）。

---

## 必读上下文与技能

在每次迭代开始前，必须：

1. 阅读或复习：
   - `CLAUDE.md` 与 `AGENTS.md`（理解禁止动真实网络栈等约束）。
   - `docs/tui/reading-notes.md`
   - `docs/tui/the-tui-commandments.md`
   - `docs/tui/beautiful-terminal-uis-python.md`
   - `docs/tui/inclusive-cli-best-practices.md`
   - `docs/tui/julia-evans-terminal-rules.md`
2. 应用 `tui-cli-design` skill，并遵循其中给出的流程和检查清单来设计和审查 TUI / CLI 的架构与交互。

只在需要时再查阅其他文件（如 `README.md`、`p_doc/` 下的文档）以理解当前实现和已有 UX 约定。

---

## 工作方式：一轮只做一个任务

- 将整体“重构 / 优化 UI / UX”的工作拆解成若干小而完整的子任务，每一轮 Ralph 迭代 **只完成一个子任务**。
- 使用 `docs/tui_ux_plan.md` 作为计划与进度文件：
  - 如果该文件不存在：在第一次迭代中创建它，列出高层任务清单（使用 Markdown 复选框）。
  - 每轮开始时：读取 `docs/tui_ux_plan.md`，选择一个尚未完成的任务作为本轮目标。
  - 每轮结束时：更新 `docs/tui_ux_plan.md`，勾选已完成项，记录新发现的问题、阻塞点和下一步建议。

每一轮迭代的基本流程：

1. 读取 `docs/tui_ux_plan.md` 和上述 TUI 文档 / 技能说明。
2. 选定一个具体的、可在单轮内完成的 UI / UX 子任务（例如：统一键位和帮助面板、重构状态机、抽象 theme / layout、改善某个交互流程等）。
3. 设计实现方案：遵守 `tui-cli-design` 技能和仓库开发约定。
4. 实施最小必要的代码与文档变更。
5. 运行验证命令（见下一节）并修复可修复的问题。
6. 更新 `docs/tui_ux_plan.md`（进度与下一步）。
7. 使用一次性、原子化的 commit 提交本轮所有变更（包括代码与文档），示例流程：
   - 确认变更：`git status`。
   - 暂存相关文件：`git add ...`（至少包含本轮改动的源码 + `docs/tui_ux_plan.md` 以及相关文档）。
   - 编写清晰的提交信息：例如 `git commit -m "tui: refine navigation UX for main screen"`。
   - **不要** 在循环中执行 `git push`，将推送留给人类维护者手动完成。
8. 如果所有成功条件尚未满足，则 **不要** 输出完成承诺，而是结束本轮，让下一轮继续。

## 自我验证与回归检查

在认为某轮子任务完成之前，必须执行并通过以下检查：

1. **代码级检查**
   - 优先查找并使用本仓库已有的测试 / 检查命令（例如在 `README.md`、`CLAUDE.md`、`AGENTS.md`、`Makefile`、`shellcrashgo/tests/` 中记录的命令）。
   - 只运行那些：
     - 在文档中推荐使用；
     - 明确属于“文件级 / 逻辑级验证”，不会修改 iptables/nftables/路由/系统代理 / 默认网关 / DNS 等真实网络栈设置。
   - 如确需进行真流量 / 真路由相关自测，只能通过**短生命周期的受限 Docker 测试容器**完成（默认 bridge 网络），并严格遵守 `AGENTS.md` / `CLAUDE.md` 中“禁止使用 \`--net=host\`、\`--privileged\`、\`--cap-add=NET_ADMIN\` 等会暴露宿主网络栈参数”的约束。
   - 如果没有现成命令可用，则至少：
     - 确保代码能成功编译 / 通过基础构建（例如 Go 项目的 `go build ./...` 或 `go test ./...`，前提是这类命令在本项目是常规且安全的）。

2. **交互与 UX 级检查（静态）**
   - 审查终端行为是否符合 `tui-cli-design` 与上述 TUI 文档中的原则：
     - 键位与交互是否可预测且有帮助提示（如 `q` 退出、`?` 显示帮助、`/` 搜索等）。
     - 是否避免了不必要的装饰性噪音（复杂 ASCII art、大量动画等）。
     - 对于管道 / 重定向输出是否自动降级（如关闭颜色 / 装饰）。
   - 仅在不会触碰真实网络栈的前提下，可以用最小化场景手动运行程序来观察 TUI 行为。

只有当本轮子任务通过上述验证，且不会破坏现有行为或违反仓库约定时，才可以视为本轮完成。

## 成功标准（整体验收）

整个 Ralph 任务在满足以下条件后，才算 **完全完成**，并可以输出完成承诺：

1. `docs/tui_ux_plan.md` 中列出的核心 UI / UX 重构任务全部被勾选为已完成，并且：
   - 每个任务旁都有简要说明其实现方式或链接到相关代码位置。
2. `shellcrashgo` 的终端交互符合以下标准：
   - 常用操作有清晰、统一的键位和帮助说明。
   - 界面布局与状态切换可预测，不会在相同输入下产生不确定布局。
   - 主题 / 布局 / 键位具备合理的可配置性（通过配置文件或环境变量）。
   - 对非 TTY 输出自动收敛为简洁、可日志化 / 可被脚本消费的格式（必要时提供 JSON 或“简单模式”）。
3. 所有与 UI / UX 相关的更改均通过了仓库推荐的安全自动化检查（构建 / 测试 / Lint 等），且：
   - 没有引入编译错误或测试回归。
4. 未违反 `CLAUDE.md` / `AGENTS.md` 中关于网络栈和环境安全的任何约束。
5. 已为主要 UI / UX 决策更新或新增文档（例如在 `docs/` 或 `p_doc/` 中），使人类维护者能理解新设计。

当且仅当上述条件全部满足时：

- 才能输出完成承诺：`COMPLETE`
- 否则，不要输出 `COMPLETE`，而是结束当前迭代，等待下一轮继续。

## 失败恢复与停机条件

如果在 **连续 5 轮迭代** 内仍无法满足所有成功标准，或遇到难以跨越的阻塞（例如：
缺乏明确需求、现有实现与预期严重冲突、需要人工决策等），则：

1. 在 `docs/tui_ux_plan.md` 中追加一个“阻塞 / 后续工作”章节，至少包含：
   - 目前尚未完成的任务列表。
   - 每个任务被阻塞的原因（技术障碍 / 需求不清 / 风险过大等）。
   - 已尝试过的方案及其结果。
   - 你建议人类维护者在后续手动决策的选项与权衡。
2. 确保所有已完成的子任务仍然通过验证命令，没有引入新的错误。
3. **不要** 输出 `COMPLETE`，而是明确在计划文件中说明“任务未完全完成，但已达到自动化可行的上限”。

## 输出约定与注意事项

- 每一轮迭代的主要成果应体现在：
  - 代码与配置的变更；
  - `docs/tui_ux_plan.md` 的更新（任务状态、备注、下一步建议）；
  - 必要时补充 / 更新 `docs/` 或 `p_doc/` 下的文档说明。
- 只有在**所有成功标准都达成并通过验证**后，才可以在输出中包含字面量 `COMPLETE` 作为完成承诺。否则严禁输出该词，以免 Ralph 过早停止。
